## Запити зацікавлених осіб

### Вступ

У цьому документі зібрано усі вимоги й очікування різних груп користувачів щодо системи аналізу медіаконтенту. Зацікавленими сторонами можуть бути комерційні компанії, громадські організації, політичні партії, маркетингові агентства, а також публічні особи та знаменитості.

### Мета

Головна мета платформи — відстежувати суспільні настрої стосовно подій, брендів або політичних кампаній. Отримана аналітика допомагає ухвалювати стратегічні рішення та протидіяти фейкам і дезінформації.

### Контекст

Документ описує сценарії використання системи, їх учасників, передумови, очікувані результати та можливі виняткові ситуації. Крім того, наведено ролі користувачів, функціональність і загальні вимоги до платформи.

### Основні терміни

| Термін | Визначення |
|--------|------------|
| **Медіаконтент** | Інформаційне наповнення друкованих, аудіовізуальних, електронних або конвергентних медіа у вигляді тексту, зображення чи аудіо, орієнтоване на масову аудиторію. |
| **Аналітика** | Логіко‑інтелектуальна діяльність, спрямована на вирішення практичних завдань. |
| **Обліковий запис** | Стандартизований спосіб надання користувачеві доступу до інформаційної служби чи мережі. |
| **Ідентифікатор** | Унікальне ім’я об’єкта програми, яке дозволяє звертатися до нього у визначеному контексті. |
| **Автоматизація** | Застосування технічних і програмних засобів для мінімізації участі людини в обробці інформації. |
| **Валідація** | Перевірка готового ПЗ на відповідність вимогам і очікуванням замовника. |

## 1. Бізнес‑процеси <a id="bp"></a>

Ефективність моделі аналізу медіаконтенту залежить від інтеграції платформи у внутрішнє середовище організації та розуміння зовнішніх і внутрішніх чинників.

### Внутрішні ролі

* **Аналітик‑координатор** — формує методику досліджень, ставить завдання й відповідає за якість результатів.
* **Дослідник** — безпосередньо аналізує контент, використовуючи спеціалізовані інструменти.
* **Технічний експерт** — підтримує працездатність системи та слідкує за безпекою.

### Зовнішні ролі

* **Замовник** — ініціює аналіз і отримує результати через команду виконавців.

### Сценарії

#### UserReg — Реєстрація користувача

| Поле | Значення |
|------|----------|
| **Учасники** | Користувач, Система |
| **Передумови** | Користувач іще не має облікового запису |
| **Результат** | Створено обліковий запис |
| **Виключення** | `AlreadyRegistered`, `DataMissing`, `InvalidData` |
| **Кроки** | 1. Відкрити форму реєстрації.<br>2. Заповнити дані (ім’я, email, пароль тощо).<br>3. Система перевіряє унікальність email.<br>4. Система створює обліковий запис.<br>5. Користувач отримує підтвердження. |

#### UserAuth — Авторизація користувача

| Поле | Значення |
|------|----------|
| **Учасники** | Користувач, Система |
| **Передумови** | Обліковий запис існує |
| **Результат** | Вхід до особистого кабінету |
| **Виключення** | `DataNotFound`, `NotRegistered`, `TooManyAttempts` |
| **Кроки** | 1. Відкрити форму логіну.<br>2. Ввести email і пароль.<br>3. Система перевіряє облікові дані.<br>4. У разі успіху надається доступ. |

#### UserDelete — Видалення користувача

| Поле | Значення |
|------|----------|
| **Учасники** | Технічний експерт, Користувач, Система |
| **Передумови** | Користувач має права на видалення |
| **Результат** | Обліковий запис видалено |
| **Виключення** | `PermissionDenied`, `NotRegistered` |
| **Кроки** | 1. Технічний експерт обирає користувача.<br>2. Система перевіряє права.<br>3. Система видаляє обліковий запис.<br>4. Підтвердження видалення. |

#### UserRolePromote — Підвищення ролі

| Поле | Значення |
|------|----------|
| **Учасники** | Технічний експерт, Користувач, Система |
| **Передумови** | Технічний експерт має права змінювати ролі |
| **Результат** | Роль користувача змінено |
| **Виключення** | `RoleNotEnoughRights`, `RoleAlreadyHighest`, `NotRegistered` |
| **Кроки** | 1. Обрати користувача.<br>2. Обрати нову роль.<br>3. Система перевіряє права експерта.<br>4. Роль оновлюється.<br>5. Підтвердження зміни. |

#### MediaFind — Пошук медіаконтенту

| Поле | Значення |
|------|----------|
| **Учасники** | Користувач, Система |
| **Передумови** | Авторизований користувач |
| **Результат** | Видача результатів аналізу |
| **Виключення** | `InformationNotFound`, `InvalidSearchRequest` |
| **Кроки** | 1. Задати фільтри та запит.<br>2. Система здійснює пошук.<br>3. Контент аналізується.<br>4. Формується звіт.<br>5. Результати виводяться користувачу. |

#### MediaCreate — Створення медіаконтенту

| Поле | Значення |
|------|----------|
| **Учасники** | Користувач, Система |
| **Передумови** | У користувача є права створювати контент |
| **Результат** | Новий контент збережено |
| **Виключення** | `PermissionDenied`, `ContentIdExists` |
| **Кроки** | 1. Обрати опцію «Створити».<br>2. Заповнити форму й додати файли.<br>3. Система перевіряє дані.<br>4. Контент зберігається.<br>5. Підтвердження створення. |

#### MediaDelete — Видалення медіаконтенту

| Поле | Значення |
|------|----------|
| **Учасники** | Користувач, Система |
| **Передумови** | Права на видалення контенту |
| **Результат** | Контент видалено |
| **Виключення** | `PermissionDenied`, `InvalidContentId` |
| **Кроки** | 1. Обрати контент.<br>2. Система перевіряє права.<br>3. Система перевіряє існування контенту.<br>4. Видалення та підтвердження. |

#### MediaEdit — Редагування медіаконтенту

| Поле | Значення |
|------|----------|
| **Учасники** | Користувач, Система |
| **Передумови** | Права на редагування контенту |
| **Результат** | Інформацію оновлено |
| **Виключення** | `PermissionDenied`, `InvalidContentData` |
| **Кроки** | 1. Обрати контент для редагування.<br>2. Внести зміни.<br>3. Система перевіряє права й дані.<br>4. Зберігає оновлення.<br>5. Підтвердження. |

#### MediaAddFeedback — Додавання відгуку

| Поле | Значення |
|------|----------|
| **Учасники** | Користувач, Система |
| **Передумови** | Авторизований користувач |
| **Результат** | Відгук додано |
| **Виключення** | `InvalidContentId`, `InvalidComment` |
| **Кроки** | 1. Обрати контент.<br>2. Ввести текст і оцінку.<br>3. Система перевіряє валідність.<br>4. Зберігає відгук.<br>5. Підтвердження. |

## 2. Огляд продукту <a id="overview"></a>

Система — це хмарний сервіс для автоматизованого аналізу текстів, зображень, аудіо й відео.

### Категорії користувачів

* **Медіакомпанії** — відстеження реакції аудиторії та трендів.
* **Маркетингові агентства** — моніторинг брендів і ефективності кампаній.
* **Дослідницькі центри** — контент‑аналіз у наукових дослідженнях.
* **Бренди та компанії** — збір зворотного зв’язку споживачів.
* **Урядові й НГО** — моніторинг громадської думки.
* **Журналісти** — пошук матеріалів і аналіз відгуків.
* **SMM‑менеджери** — оптимізація контенту в соціальних мережах.
* **Служби підтримки** — оперативне реагування на запити клієнтів.

## 7. Експлуатаційна придатність <a id="maintainability"></a>

Експлуатаційна придатність визначає, наскільки легко систему підтримувати, оновлювати та масштабувати після впровадження.

### Ключові вимоги

* **Сучасний стек технологій** — використання актуальних фреймворків і бібліотек, що мають активну спільноту та регулярні оновлення.
* **Регулярні оновлення** — планові патч‑релізи безпеки й функціоналу, мінімальний час простою при розгортанні.
* **Модульна архітектура** — можливість незалежно оновлювати компоненти, спрощуючи інтеграцію нових функцій.
* **Документований API** — REST/GraphQL‑ендпоїнти з auto‑generated документацією (OpenAPI/Swagger), що полегшує інтеграцію з іншими сервісами.
* **CI/CD‑конвеєр** — автоматичні тести, статичний аналіз коду й деплой у кілька кліків.
* **Рефакторинг коду** — регулярний перегляд і оптимізація, аби зменшити технічний борг.
* **Інструменти моніторингу** — логування, алерти та дашборди (Prometheus + Grafana) для швидкого виявлення та усунення проблем.
* **Спрощене розгортання** — контейнеризація (Docker) та оркестрація (Kubernetes) для гнучкого масштабування.
* **Підготовка персоналу** — детальна технічна документація й навчальні матеріали для адміністраторів і користувачів.
